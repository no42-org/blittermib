# MIB Corpus

This directory holds the curated MIB collection that ships with
blittermib. Vendor MIBs are organised by IANA Private Enterprise
Number; IETF MIBs are bucketed into functional groups.

## Layout

```
mibs/
├── ietf/
│   ├── core/         # SNMPv2-SMI, SNMPv2-TC, SNMP-MIB, ...
│   ├── transport/    # TCP-MIB, UDP-MIB, IP-MIB, IPV6-MIB, ...
│   ├── interfaces/   # IF-MIB, EtherLike-MIB, ...
│   ├── routing/      # OSPF-MIB, BGP4-MIB, RIP2-MIB, ...
│   ├── monitoring/   # RMON-MIB, RMON2-MIB, HOST-RESOURCES-MIB, ...
│   ├── management/   # ENTITY-MIB, DISMAN-EVENT-MIB, ...
│   ├── security/     # USM-MIB, VACM-MIB, TLS-MIB, ...
│   ├── application/  # APPLICATION-MIB, NETWORK-SERVICES-MIB, ...
│   └── other/        # default for unclassified RFC MIBs
├── iana/             # IANA-maintained registry MIBs (IANAifType-MIB, ...)
├── vendors/
│   ├── 9-cisco/
│   ├── 22610-a10/
│   ├── 2636-juniper/
│   └── ...           # one dir per vendor: {PEN}-{slug}
├── INDEX.yaml        # generated catalog (regenerated by `make index`)
├── _groups.yaml      # IETF functional-group map (hand-maintained)
├── _overrides.yaml   # per-module license overrides (hand-maintained)
└── LICENSES/         # license-text references for the tags in INDEX.yaml
```

The `{PEN}-{slug}/` form sorts numerically and gives PR reviewers
a clean check: "this MIB's `MODULE-IDENTITY` OID starts with
`.1.3.6.1.4.1.{PEN}` — does the directory match?"

## Generated vs hand-maintained

| File                  | Origin                                           |
|-----------------------|--------------------------------------------------|
| `INDEX.yaml`          | generated by `make index`; commit the diff      |
| `_groups.yaml`        | hand-maintained — append when an IETF MIB lands |
| `_overrides.yaml`     | hand-maintained — empty unless the auto-tagger needs help |
| `LICENSES/*.txt`      | hand-maintained — one per tag in `INDEX.yaml`'s license set |

`make index` is **idempotent on a stable corpus**: two consecutive
runs with no source changes produce the same byte output. The
generator preserves `added_in` dates from the prior `INDEX.yaml`
across regenerations — so a file deleted out-of-band (`rm INDEX.yaml`
followed by `make index`) will reset every entry's date to today.
If you ever need to fully rebuild, do it deliberately and explain
the date churn in the PR.
